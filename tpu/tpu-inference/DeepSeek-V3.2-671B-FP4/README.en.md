# DeepSeek V3.2 671B FP4 Inference on TPU v7x-8

> 🌐 **Languages** | **语言**: [中文](README.md) · **English**

> End-to-end guide: Run DeepSeek V3.2 671B (FP4 quantized) inference on a single TPU v7x-8 node,
> covering environment setup, weight cache generation, FP4 conversion, vLLM service launch, and accuracy validation.
>
> **Code repository**: https://github.com/yangwhale/tpu-inference (branch: `feature/moe-fp4-weight-cache`)
>
> **Model**: [deepseek-ai/DeepSeek-V3.2](https://huggingface.co/deepseek-ai/DeepSeek-V3.2) (FP8 weights)
>
> **Relationship to DeepSeek R1**: V3.2 shares the exact same architecture as R1 (671B MoE, 256 experts top-8, 61 layers, MLA).
> The inference pipeline (FP4 cache, environment variables, vLLM commands) is identical. Key difference: V3.2's thinking mode is
> **on-demand** (enabled via `chat_template_kwargs: {"thinking": true}`), while R1 **always enables** thinking by default.
> V3.2 also introduces **DSA (DeepSeek Sparse Attention)** for optimized long-context inference.

### MoE Cache Generation Strategies

There are three ways to generate FP4 MoE cache:

| Method | Steps | Duration | Storage Need | Recommended |
|------|------|------|----------|------|
| **CPU Parallel Direct** ⭐ | safetensors → FP4 (pure CPU, 12 parallel) | **~28 min** | Model 700G + FP4 540G = **~1.2 TB** | ✅ |
| TPU FP4 Direct | safetensors → FP4 (TPU JIT, sequential) | ~87 min | Model 700G + FP4 610G = **~1.3 TB** | Alternative |
| FP8→FP4 Two-step | safetensors → FP8 cache → FP4 conversion | ~60 min | Model 700G + FP8 404G + FP4 610G = **~1.7 TB** | Alternative |

**CPU Parallel Direct** (`gen_fp4_cache_cpu_parallel.py`) is the fastest method: pure numpy implementation, no TPU/JAX needed,
uses ProcessPoolExecutor with 12 workers to process 58 layers of MoE experts in parallel.
Model is read from disk, leaving all RAM for the FP32 intermediate arrays during conversion.

TPU FP4 direct conversion is achieved by setting `MOE_REQUANTIZE_WEIGHT_DTYPE=float4_e2m1fn`, skipping the FP8 intermediate step and saving ~400 GB storage.

> **Cache subdirectory name** is auto-generated by `_get_config_cache_subdir()`, format `ep{EP}_tp{TP}_{backend}_{dtype}_bs{block}`.
> Currently validated configurations all use backend `gmm_ep` (EP=8, TP=1), with subdir names like `ep8_tp1_gmm_ep_fp4e2m1_bsNone`.

# Part B: TPU VM Bare Metal Installation

For direct development/testing on TPU VM, without Docker.

## B-1: Create TPU VM + Data Disk

```bash
export PROJECT=<PROJECT_ID>
export ZONE=us-central1-c
export TPU_NAME=my-deepseek-vm

# Create TPU VM (note scopes and oslogin)
gcloud alpha compute tpus queued-resources create ${TPU_NAME}-qr \
  --node-id $TPU_NAME \
  --project $PROJECT \
  --zone $ZONE \
  --accelerator-type tpu7-8 \
  --runtime-version v2-alpha-tpu7-ubuntu2404 \
  --scopes=https://www.googleapis.com/auth/cloud-platform \
  --metadata=enable-oslogin=false \
  --service-account <SA_EMAIL>

# Wait for state to become ACTIVE (~3-5 min)
watch -n 10 gcloud alpha compute tpus queued-resources describe ${TPU_NAME}-qr \
  --project $PROJECT --zone $ZONE --format="value(state.state)"

# Create 2TB Hyperdisk Balanced data disk (sequential R/W 2.4 GB/s)
gcloud compute disks create ${TPU_NAME}-data \
  --project $PROJECT --zone $ZONE \
  --type hyperdisk-balanced \
  --size 2TB \
  --provisioned-iops 40000 \
  --provisioned-throughput 2400

# Attach data disk to TPU VM
gcloud alpha compute tpus tpu-vm attach-disk $TPU_NAME \
  --disk ${TPU_NAME}-data \
  --zone $ZONE --project $PROJECT --mode read-write

# SSH connection
gcloud compute tpus tpu-vm ssh $TPU_NAME --project $PROJECT --zone $ZONE
```

After entering the VM, format and mount the data disk:

```bash
# Data disk is usually /dev/nvme1n1
lsblk | grep nvme

# Format + mount
sudo mkfs.ext4 -m 0 -E lazy_itable_init=0,lazy_journal_init=0 /dev/nvme1n1
sudo mkdir -p /data
sudo mount -o discard,defaults /dev/nvme1n1 /data
sudo chown $USER:$USER /data

# Add to fstab (auto-mount on reboot)
echo "/dev/nvme1n1 /data ext4 discard,defaults 0 2" | sudo tee -a /etc/fstab

# Expand /dev/shm (default is half of memory, not enough for FP4 cache)
sudo mount -o remount,size=800G /dev/shm
df -h /dev/shm
# Expected: 800G
```

## Step 4: Generate FP4 MoE Cache (Recommended: CPU Parallel Direct)

> **Recommended approach**: Pure CPU numpy implementation (`gen_fp4_cache_cpu_parallel.py`), no TPU/JAX required,
> 12 workers parallel, 58 layers in just **~28 min**. Direct safetensors-to-FP4 conversion, no intermediate FP8 cache needed.

```bash
# Ensure vllm_env has ml_dtypes, safetensors, torch
source ~/vllm_env/bin/activate

# Confirm model and storage
ls $MODEL/*.safetensors | wc -l   # Expected: 163
df -h $STORAGE                     # Need ~620 GB available space

# Get gen script (in same directory as this README)
# GKE:  cp $TI_DIR/../gpu-tpu-pedia/.../gen_fp4_cache_cpu_parallel.py /tmp/
# TPU VM: run directly from repo
export GEN_SCRIPT=$TI_DIR/../gpu-tpu-pedia/tpu/tpu-inference/DeepSeek-V3.2-671B-FP4/gen_fp4_cache_cpu_parallel.py
# If gpu-tpu-pedia not cloned, download separately:
# curl -LO https://raw.githubusercontent.com/yangwhale/gpu-tpu-pedia/main/tpu/tpu-inference/DeepSeek-V3.2-671B-FP4/gen_fp4_cache_cpu_parallel.py
# export GEN_SCRIPT=./gen_fp4_cache_cpu_parallel.py

# Launch CPU parallel FP4 cache generation
# --workers: parallelism, adjust based on available RAM (each worker peaks ~70 GB)
# E.g., 944 GB RAM v7x-8 machine can run 12 workers
python3 -u $GEN_SCRIPT \
  --model-dir $MODEL \
  --cache-dir $STORAGE/moe-cache/ep8_tp1_gmm_ep_fp4e2m1_bsNone \
  --workers 12

# Resume after interrupt (auto-skips completed layers, no --force needed)
```

**How it works**:

```
safetensors (FP8 e4m3fn)
    │
    │  load_layer_experts(): load 256 experts' gate/up/down weights layer by layer
    ▼
FP8 weights (256, 4096, 7168) + block scale (256, 32, 56)
    │
    │  dequant_fp8_blocked(): 128×128 block dequantization (reshape → broadcast multiply)
    ▼
FP32 weights (256, 4096, 7168)
    │
    │  quantize_to_fp4(): per-channel quantization (axis=2, abs_max / fp4_max)
    ▼
FP4 weights (256, 4096, 7168) + scale (256, 4096, 1)
    │
    │  GMM_EP layout: swapaxes(1,2) + expand_dims
    ▼
FP4 cache npy:
  w13_weight:       (256, 7168, 4096) float4_e2m1fn
  w13_weight_scale: (256, 1, 1, 4096) float32
  w2_weight:        (256, 2048, 7168) float4_e2m1fn
  w2_weight_scale:  (256, 1, 1, 7168) float32
```

**Key Design Decisions**:

| Decision | Reason |
|------|------|
| ProcessPoolExecutor (not Thread) | OS fully reclaims memory on process exit, avoiding glibc malloc arena fragmentation OOM |
| max_tasks_per_child=1 | Kill and rebuild worker process after each layer to ensure no memory leak |
| in-place numpy operations | `w_fp32 *= scale_inv`, `np.clip(..., out=)` avoid allocating large temp arrays |
| w13/w2 sequential processing | Complete w13 full pipeline and release memory, then process w2, reducing peak from ~130GB to ~70GB/worker |
| Read model from disk (not /dev/shm) | Free 800G RAM for workers, load adds ~20s/layer but allows more parallelism |

**Worker count selection**:

| Machine RAM | Recommended workers | Rationale |
|----------|-------------|------|
| 944 GB (v7x-8) | 12 | 800G available ÷ 70G peak/worker |
| 500 GB | 6 | Reserve 80G for system |
| 250 GB | 3 | Minimum config |

## Step 5: Pre-copy FP4 Cache + Non-MoE Weights to /dev/shm (Highly Recommended)

**Strongly recommended** to copy cache and non-MoE weights to `/dev/shm` (tmpfs). Beyond accelerating startup, this also **avoids vLLM's MoE prefetch deadlock** (see [Pitfall #13](#13-vllm-moe-prefetch-deadlock-cache-hit-path)).

### Step 5a: Copy FP4 cache

```bash
# FP4 cache is ~610 GB, /dev/shm needs sufficient space
df -h /dev/shm
# TPU VM default /dev/shm = half of host memory (v7x-8 ~473 GB)
# GKE Pod can configure emptyDir.sizeLimit: 800Gi

# If space sufficient:
time cp -r $STORAGE/moe-cache/ep8_tp1_gmm_ep_fp4e2m1_bsNone /dev/shm/
# Hyperdisk Balanced 2TB: ~27 min (~383 MB/s)
# Hyperdisk Extreme 4TB: ~12 min (~850 MB/s)

# Subsequent startup points to /dev/shm
export MOE_WEIGHT_CACHE_DIR=/dev/shm
```

### Step 5b: Extract and Copy Non-MoE Weights (Optional, Recommended)

Extract non-MoE weights from 163 safetensors shards into a single file, place in `/dev/shm` cache directory.
vLLM auto-detects and loads on startup, skipping 70-shard scan, **weight loading from 4:26 down to 21s**.

```bash
# Extract non-MoE weights (~23 GB, takes ~3 min)
# Script in same directory as this README
python3 extract_non_moe_weights.py \
  --model-dir $MODEL \
  --output $STORAGE/moe-cache/ep8_tp1_gmm_ep_fp4e2m1_bsNone/non_moe_weights.safetensors
# If script not local, download:
# curl -LO https://raw.githubusercontent.com/yangwhale/gpu-tpu-pedia/main/tpu/tpu-inference/DeepSeek-V3.2-671B-FP4/extract_non_moe_weights.py

# Copy to /dev/shm (if Step 5a already copied cache, only need to copy this one file)
cp $STORAGE/moe-cache/ep8_tp1_gmm_ep_fp4e2m1_bsNone/non_moe_weights.safetensors \
   /dev/shm/ep8_tp1_gmm_ep_fp4e2m1_bsNone/

# Verify
ls -lh /dev/shm/ep8_tp1_gmm_ep_fp4e2m1_bsNone/non_moe_weights.safetensors
# Expected: ~22 GB
```

> **How it works**: vLLM startup `_filtered_safetensors_iterator()` searches `MOE_WEIGHT_CACHE_DIR`
> for `non_moe_weights.safetensors`. Once found, it directly loads 1367 non-MoE tensors from this single file,
> skipping the scan and filtering of 163 safetensors shards.
>
> **Space budget**: FP4 cache 610 GB + non-MoE 23 GB = 633 GB, /dev/shm 800 GB has 167 GB remaining.

> **TPU VM Note**: Default /dev/shm is ~473 GB, can't fit 610 GB FP4 cache.
> Can be enlarged via `sudo mount -o remount,size=800G /dev/shm`, but compresses available memory.
> If shm space insufficient, loading directly from disk also works (slower).

#### Step 4b: FP8 → FP4 Offline Conversion

FP4 halves MoE weight volume. Pure CPU computation, no TPU needed.

```bash
python3 $TI_DIR/scripts/convert_fp8_to_fp4.py \
  --fp8-cache $MOE_WEIGHT_CACHE_DIR/ep8_tp1_gmm_ep_fp8e4m3_bsNone \
  --fp4-cache $MOE_WEIGHT_CACHE_DIR/ep8_tp1_gmm_ep_fp4e2m1_bsNone \
  --workers 10
```

## Performance Data

The following data is measured on TPU v7x-8 single node, covering both TPU VM bare metal and GKE scenarios.

### Weight Loading Time (cache + consolidated non-MoE both in /dev/shm)

| Phase | Before Optimization | After (warm) | After (cold) | Notes |
|------|--------|---------------|---------------|------|
| JAX init + model config | ~65s | ~65s | ~65s | mesh creation, config parsing |
| Non-MoE weight loading | 4:26 (70 shards) | **21s** | **25-108s** | warm: pjit cache hit; cold: 25 unique shapes need recompilation |
| MoE Cache mmap → TPU | 1:22 | **21s** | **63s** | /dev/shm tmpfs; cold has additional pjit compilation overhead |
| Server init | ~22s | ~2s | ~2s | |
| **Application startup complete** | **~7 min** | **~1:48** | **~3:17-3:51** | warm requires same-process consecutive starts (see [Pitfall #20](#20-pjit-compiled-transfer-cache-not-persistent)) |

> **warm vs cold start**:
> - **warm** (~1:48): Same process consecutive restarts, pjit compiled transfer programs still in memory, weight transfer skips compilation
> - **cold** (~3:17-3:51): pjit cache lost after process restart, all 25 unique tensor shapes require recompilation (~22s). **This is the common scenario**
>
> **Two key optimizations** (see [Pitfall #18](#18-jaxclear_caches-performance-bug--commenting-one-line-10x-faster) and [Pitfall #19](#19-non-moe-weight-consolidated-file-optimization)):
> 1. Comment out `jax.clear_caches()`: non-MoE loading from 216s → **21-108s** (2-10x, depending on cold/warm)
> 2. Consolidated non-MoE file: skip 70-shard scan, single-file direct load

### MoE Cache Size

| Format | Disk Size | HBM Usage/device | Notes |
|------|----------|-----------------|------|
| FP8 cache | ~404 GB | ~101.8 GB (exceeds HBM) | 58 layers × ~7 GB/layer |
| FP4 cache | ~610 GB | ~60.9 GB | 58 layers × ~10.5 GB/layer (incl. FP32 scale) |

> **Note**: FP4 cache files are larger than FP8 because native FP4 storage requires additional scale precision (FP32 scale).
> But after loading to TPU HBM, FP4 weights occupy half the memory of FP8.
> When using FP4 direct conversion, no FP8 cache is needed.

### FP4 Cache Generation

| Method | Duration | Per Layer | Concurrency | Notes |
|------|------|------|------|------|

| vLLM online generation | ~5 h | ~5 min | 1 | safetensors full load + requant |
| FP8→FP4 two-step | ~60 min | - | 1 | FP8 generation ~30 min + FP4 conversion ~30 min |

> Model read from disk (not /dev/shm), load takes ~7-55s/layer (depending on page cache), all RAM left for computation.

## FAQ

### Q: Why not use FP8 directly, why convert to FP4?

FP8 MoE weights need ~101.8 GB HBM per device, exceeding v7x single device's 94.75 GB limit.
FP4 halves MoE weights to ~60.9 GB/device; combined with attention weights, total ~70 GB/device, leaving sufficient KV cache space.

### Q: Will FP4 conversion lose accuracy?

Conversion uses dequant→rescale→requant pipeline (not simple truncation):
1. FP8 × scale → FP32 (recover true value)
2. Compute new per-channel scale = abs_max / 6.0 (FP4 max)
3. FP32 / new_scale → clip → FP4

GSM8K test verifies whether accuracy is within acceptable range.

### Q: Can I delete FP8 cache after first launch?

Yes. FP4 cache contains all needed information. But recommend keeping it for fallback.

### Q: TPU VM /dev/shm not large enough, what to do?

TPU VM default /dev/shm = half of host memory (v7x-8 ~473 GB), can't fit 610 GB FP4 cache. Two solutions:
1. `sudo mount -o remount,size=800G /dev/shm` to expand (compresses available memory)
2. Load directly from disk (no shm), per-layer load time goes from ~1.3s to ~28s, total ~25 minutes slower

### Q: What's the difference between GKE and TPU VM?

| Comparison | GKE + Docker | TPU VM Bare Metal |
|------|-------------|-------------|
| Environment isolation | Docker container, consistent env | Direct system Python |
| /dev/shm | Configurable `emptyDir.sizeLimit: 800Gi` | Default 473 GB, manual expansion needed |
| Storage | Lustre PVC or PD | GCS/PD/Lustre self-mounted |
| Use case | Production deployment | Development debugging, quick testing |

### Q: How to choose between PyPI install and source install?

- **PyPI (`pip install vllm-tpu`)**: Simplest, but FP4 branch changes not yet released to PyPI
- **Source install**: Currently the only way, because we need `feature/moe-fp4-weight-cache` branch changes
- After FP4 merges to mainline, can directly use `pip install vllm-tpu`

### Q: Can it run directly on bare metal (TPU VM)?

**Yes.** Both bare metal and Docker have been verified. Key considerations:

1. **cd /tmp**: Must launch from non-`~/vllm` directory, otherwise Python treats `~/vllm/` as namespace package
2. **JAX version**: After installing vLLM, must manually upgrade JAX to 0.9.2 (see [Pitfall #1](#1-jax-version-downgraded-by-requirementstputxt))
3. **`from vllm import SamplingParams` errors**: This is a circular import bug, but doesn't affect `python3 -m vllm.entrypoints.openai.api_server`

### Q: Cache generation less than 58 layers, what to do?

vLLM process may exit early or async cache write may not complete. Already-generated cache is intact; use `gen_missing_fp4_cache.py` to fill in:

**Critical: Weights must not be transposed; preserve safetensors original shape.** `process_fp8_moe_weights` internal convention:
- w13 = `(E, 2*intermediate, hidden)` = (256, 4096, 7168) — gate+up concat on axis=1
- w2 = `(E, hidden, intermediate)` = (256, 7168, 2048)

```python
# gen_missing_fp4_cache.py — key part
os.environ["MOE_REQUANTIZE_WEIGHT_DTYPE"] = "float4_e2m1fn"  # Must set before import

# Don't transpose! safetensors original layout is correct:
gate_stack = torch.stack([gate_w[i] for i in range(E)])   # (E, 2048, 7168)
up_stack = torch.stack([up_w[i] for i in range(E)])       # (E, 2048, 7168)
w13_weight = torch.cat([gate_stack, up_stack], dim=1)     # (E, 4096, 7168)
w2_weight = torch.stack([down_w[i] for i in range(E)])    # (E, 7168, 2048)
```

Full script in [`gen_fp4_cache_optimized.py`](gen_fp4_cache_optimized.py) in this directory (with prefetch + async save optimizations).

```bash

cd /tmp && source ~/vllm_env/bin/activate
python3 gen_fp4_cache_optimized.py --backend gmm_ep

# Specify custom path
python3 gen_fp4_cache_optimized.py --backend gmm_ep --cache-dir /data/moe-cache/ep8_tp1_gmm_ep_fp4e2m1_bsNone

# Force regenerate all layers
python3 gen_fp4_cache_optimized.py --force
```

Old version [`gen_missing_fp4_cache.py`](gen_missing_fp4_cache.py) still usable (sequential version, for reference).

## Pitfall Log

> The following pitfalls were discovered during DeepSeek R1 671B FP4 inference testing. Since V3.2 shares
> the exact same architecture, all issues and solutions apply equally to V3.2.

### 1. JAX Version Downgraded by requirements/tpu.txt

**Symptom**: After `uv pip install -e .` (vLLM), JAX downgraded from 0.9.2 to 0.8.0, TPU v7 init fails (`No ba16c7433 device found`)

**Cause**: `requirements/tpu.txt` depends on `tpu-inference==0.12.0` (PyPI), which pulls in old JAX 0.8.0

**Fix**:
```bash
uv pip install jax==0.9.2 jaxlib==0.9.2 libtpu==0.0.39 flax==0.12.4
```
**Lesson**: After every `pip install -e .`, check `python3 -c "import jax; print(jax.__version__)"`

### 2. flax Version Incompatible

**Symptom**: `cannot import name 'mutable_array' from 'jax._src.core'`

**Cause**: Old flax (0.10.x/0.11.x) uses internal API removed in JAX 0.9.2

**Fix**: `uv pip install flax==0.12.4` (consistent with Docker image)

### 3. `_load_weights` shape mismatch (Upstream Bug)

**Symptom**: `weights.shape=(256, 2048, 7168) param.value.shape=(256, 7168, 2048)`

**Cause**: `jax_array_from_reshaped_torch` only auto-transposes 2D tensors (`permute_dims=(1,0)`). After adding expert dim `reshape_dims=(1,) + shape` it becomes 3D, skipping the transpose logic

**Impact**: First FP8 cache generation Docker process triggers shape mismatch crash when loading non-MoE weights after processing 54/58 layers MoE cache

**Workaround**:
- Use Docker for FP8 cache generation (cache generated during MoE processing phase, unaffected by this bug)
- Fill in missing 4 layers after crash with `gen_missing_cache.py`

### 4. Weight Transpose Direction

**Symptom**: `quantize_tensor` reports `axis=2 of tensor.shape=(256, 7168, 6144) is not divisible by block=4096`

**Cause**: When manually concatenating MoE weights, incorrectly applied `.T` (transpose) to gate/up, causing w13 shape to go from correct `(E, 4096, 7168)` to `(E, 7168, 4096)`. `quantize_moe_weights` padding logic does incorrect alignment on reversed dims

**Lesson**: safetensors weight format (gate/up: `(intermediate, hidden)`, down: `(hidden, intermediate)`) already matches `process_fp8_moe_weights` input convention. **No transpose needed**. Key code comment in `fp8.py:1013-1016`

### 5. Docker Requires `--net=host`

**Symptom**: `Failed to connect to [::]:8353` (TPU runtime port)

**Cause**: TPU runtime uses fixed ports for communication; Docker default network isolation blocks the connection

**Fix**: `docker run --privileged --net=host`

### 6. Docker-Written Files Owned by root

**Symptom**: FP8 cache files generated inside Docker have root owner; bare metal Python can't write to same directory

**Fix**: `sudo chown -R $USER:$USER /data/moe-cache/`

### 7. vLLM namespace package Conflict

**Symptom**: `vllm.__path__` points to `~/vllm/` (git repo root) instead of editable install in site-packages

**Cause**: When Python launches from `~/`, `~/vllm/` is recognized as namespace package, overriding the correct import path

**Fix**: Before launching vLLM, `cd /tmp` (or any location not containing `vllm/` subdirectory)

### 8. libtpu lockfile Residue

**Symptom**: `ABORTED: Internal error when accessing libtpu multi-process lockfile`

**Cause**: Previous JAX process exited abnormally, lockfile not cleaned up

**Fix**: `sudo rm /tmp/libtpu_lockfile`

### 9. TPU device busy

**Symptom**: `TPU initialization failed: open(/dev/vfio/4): Device or resource busy`

**Cause**: Another process (Docker container or Python process) is using TPU

**Fix**: `sudo docker stop <container>` or `kill <pid>`, then `sudo rm /tmp/libtpu_lockfile`

### 10. Async Cache Write Incomplete

**Symptom**: After `Application startup complete`, cache directory only has 15/58 layers and stops growing

**Cause**: vLLM's MoE cache write is asynchronous (background thread). If process receives inference request or is `Ctrl+C` interrupted, async writer may be blocked or terminated early

**Fix**:
- After `Application startup complete`, **don't immediately send inference requests**, wait for cache to finish writing
- Use `ls ... | wc -l` to monitor until all 58 layers arrive
- If process exited causing incomplete cache, use `gen_missing_fp4_cache.py` to fill in

### 11. max-model-len 128 Too Small for Chat API

**Symptom**: Chat Completions API returns `maximum context length is 128 tokens ... your prompt contains 62 characters`

**Cause**: `--max-model-len 128` is mainly for cache generation (minimizing KV cache memory). Chat API's chat template adds system tokens, easily exceeding 128

**Fix**: Use `--max-model-len 128` for cache generation, `--max-model-len 4096` or larger for actual inference

### 12. FP4 Direct Conversion Doesn't Trigger Shape Mismatch Bug

**Discovery**: With FP4 direct conversion (`MOE_REQUANTIZE_WEIGHT_DTYPE=float4_e2m1fn`), all 58 layers (including 57-60) process normally, doesn't trigger `_load_weights` shape mismatch bug from Pitfall #3

**Cause**: FP4 direct conversion executes requantization in `process_weights_after_loading`, where weights have already been correctly reshaped. Shape mismatch occurs in subsequent `_load_weights` when loading non-MoE weights to JAX array, but FP4 direct conversion's code path completes MoE cache save before this point

**Significance**: This is another reason FP4 direct conversion is superior to two-step method — not only simpler, also bypasses known bug

### 13. vLLM MoE prefetch deadlock (cache hit path)

**Symptom**: When loading existing FP4 cache from disk, vLLM permanently hangs after loading 8th layer. All 1600+ threads blocked in `futex_wait_queue`, RSS doesn't grow. Reproduced 3 times.

**Cause**: `fp8.py`'s prefetch mechanism uses `Semaphore(_MOE_PREFETCH_AHEAD=8)` to control concurrency:
1. `_start_moe_prefetch()` submits all 58 cache files to thread pool
2. First 8 prefetch threads acquire semaphore successfully, load cache
3. Subsequent threads block on `acquire()` waiting for permits to release
4. `_get_prefetched_cache()` releases() only after main thread consumes
5. But main thread is blocked on `block_until_ready()` waiting for TPU DMA, can't call `_get_prefetched_cache()`
6. → Classic circular wait deadlock

```
prefetch threads → acquire() → blocked (8 permits exhausted)
        ↑                              ↓
main thread → block_until_ready() → can't call _get_prefetched_cache() → no release()
```

**Fix**: Place cache in `/dev/shm` (tmpfs), prefetch threads complete instantly without semaphore blocking, breaking the deadlock. Or use independent gen script to generate directly to `/dev/shm`

### 14. NEW_MODEL_DESIGN=1 Required

**Symptom**: Without `NEW_MODEL_DESIGN=1`, vLLM startup fails with: `MLA models require both the NEW_MODEL_DESIGN=1 environment variable...`

**Cause**: vLLM code update, MLA architecture models (DeepSeek V3/V3.2/R1) mandate `NEW_MODEL_DESIGN=1` + DP attention

**Fix**: Set `NEW_MODEL_DESIGN=1` and add DP attention parameters to `--additional_config`. Backend remains `GMM_EP` (EP=8, TP=1)

### 15. EngineCore Orphan Process Continues Holding TPU

**Symptom**: After killing vLLM main process, TPU still "device busy". `fuser /dev/vfio/*` shows EngineCore subprocess alive

**Cause**: vLLM V1 engine spawns EngineCore subprocess via `multiprocessing`. `kill` on main process doesn't necessarily signal subprocess

**Fix**:
```bash
# Method 1: Kill EngineCore directly
ps aux | grep EngineCore | grep -v grep
kill -9 <EngineCore PID>

# Method 2: Use fuser to find and kill all VFIO holders (recommended)
fuser /dev/vfio/4              # See who's using
fuser -k /dev/vfio/*           # Kill all VFIO holders (needs root or privileged)

# Clean up lockfile and IPC files
rm -f /tmp/libtpu_lockfile /tmp/.vllm_ipc_*
```

**Lesson**: After every vLLM stop, must use `fuser /dev/vfio/*` to confirm TPU device released. Just `kill` on main process insufficient; EngineCore subprocess may live in `SLl` state (not zombie, still holds VFIO fd). After `pkill -9 -f python`, must also use fuser to confirm

### 16. gen Script async save Causes HBM Leak

**Symptom**: `gen_fp4_cache_optimized.py` OOMs after processing 3-4 layers: `RESOURCE_EXHAUSTED: Attempting to allocate 66.50G`

**Cause**: async save's `ThreadPoolExecutor` holds reference to `output_weights` (FusedMoEWeights containing JAX device arrays). Python GC can't reclaim device arrays referenced in Future, HBM leaks accumulate

**Fix**: Before submitting async save, complete device→host copy on main thread (`np.asarray()`), then submit pure numpy data to save thread. JAX arrays released immediately

```python
# Wrong (leaks):
save_pool.submit(save_layer_cache, out_dir, output_weights)  # holds device refs

# Correct:
np_data = weights_to_numpy(output_weights)  # copy to host
del output_weights  # release device memory
save_pool.submit(save_layer_numpy, out_dir, np_data)  # numpy only
```

### 17. CPU Parallel FP4 Generation Memory Management (3 OOM Lessons)

**Lesson 1: Don't put model in /dev/shm and run conversion**

**Symptom**: Model 673G fills /dev/shm, remaining RAM insufficient for workers' FP32 intermediate arrays

**Fix**: Read model from disk (/data/models/), load slower ~20s/layer but frees 800G RAM for computation.
12 workers × 70G peak = 840G ≈ available RAM

**Lesson 2: glibc malloc doesn't return large blocks**

**Symptom**: ThreadPoolExecutor first 3 batches normal, 4th batch OOMs. `del array` + `gc.collect()` doesn't decrease RSS

**Cause**: glibc malloc arena fragmentation. Large FP32 arrays (30 GB) allocated by numpy don't release back to OS after free

**Fix**: Use `ProcessPoolExecutor(max_tasks_per_child=1)` — kill and rebuild worker process after each layer,
OS directly reclaims all memory. Cost is ~1s extra startup overhead per layer, negligible

**Lesson 3: numpy temp arrays must use in-place operations**

**Symptom**: Estimated peak ~50 GB per worker, actual ~130 GB

**Cause**: `np.abs(w)` → 30G copy, `w * scale_inv` → 30G, `np.clip(result)` → 30G.
quantize_to_fp4 internally creates 3 full-sized FP32 temp arrays

**Fix**:
```python
# Before optimization (3× temp arrays = 90 GB):
abs_max = np.max(np.abs(w_fp32), axis=2, keepdims=True)
fp4 = np.clip(w_fp32 * scale_inv, fp4_min, fp4_max).astype(fp4_dtype)

# After optimization (1× temp array = 30 GB):
abs_buf = np.abs(w_fp32)           # 1 copy
abs_max = np.max(abs_buf, ...)
del abs_buf                         # release immediately
w_fp32 *= scale_inv                 # in-place
np.clip(w_fp32, ..., out=w_fp32)    # in-place
fp4 = w_fp32.astype(fp4_dtype)
```

### 18. `jax.clear_caches()` Performance Bug — Commenting One Line 10x Faster

**Symptom**: Non-MoE weight loading 1367 tensors takes 216s (3.3 it/s), PCIe utilization only 0.3%

**Cause**: `assign_and_shard_param()` calls `jax.clear_caches()` after each tensor load, clearing 100 Python LRU caches + C++ pjit compiled executable cache. This causes each tensor's `jax.device_put()` to **recompile** the transfer program (even with identical shape + sharding); 1367 calls have only 25 unique shapes but recompile every time

**Fix**: Comment out line 898 of `weight_utils.py`'s `jax.clear_caches()`
```python
# weight_utils.py:assign_and_shard_param()
jax_param.value = shard_put(jax_weight, spec, mesh=param_mesh)
jax_param.set_metadata("_is_loaded", True)
del jax_weight
# jax.clear_caches()  # ← Comment this line
```

**Effect**:
| Metric | Before | After | Speedup |
|------|--------|--------|------|
| Non-MoE loading | 215.7s | **21.4s** | **10.1x** |
| Throughput | 3.3 it/s | **63.8 it/s** | **19.3x** |

**Risk Assessment**: No pitfalls. Cache stores compiled transfer programs, 25 shapes × few KB ≈ less than 1 MB, no impact on 630 GB RSS. Inference path unaffected (inference path originally has no `clear_caches()`). `clear_caches()` was originally to prevent cache bloat in dynamic shape scenarios, but weight loading has fixed shapes, this issue doesn't exist

### 19. Non-MoE Weight Consolidated File Optimization

**Symptom**: vLLM startup scans 163 safetensors shards, after filtering loads 1367 non-MoE keys from 70 shards, takes 4:26

**Cause**: Each shard requires `safe_open()` + iterate all keys + check if MoE. 70 file opens + 14336 MoE key filter checks have significant overhead

**Optimization**:
1. Use `extract_non_moe_weights.py` to extract 1367 non-MoE tensors into single `non_moe_weights.safetensors` (23 GB)
2. Modify `_filtered_safetensors_iterator()` to add Level 0 fast path: when consolidated file detected, load directly, skip all shard scanning
3. Place file in `/dev/shm` cache directory, vLLM auto-discovers on startup

**Effect** (combined with Pitfall #18 total effect): End-to-end startup from **~7 min → ~3:17 (cold) / ~1:48 (warm)**

### 20. pjit compiled transfer cache Not Persistent

**Symptom**: After vLLM process restart, startup time degrades from 1:48 to 3:17-3:51, non-MoE loading from 21s to 25-108s, MoE loading from 21s to 63s

**Cause**: `jax.device_put()` internal pjit compiled transfer programs cache in process memory (C++ LRU cache), don't persist to disk. DeepSeek V3.2/R1's 1367 non-MoE tensors only have ~25 unique (shape, sharding) combinations; warm state hits cache and skips compilation. After process restart, cache cleared, all 25 shapes recompile

**Impact**:
| Metric | warm (same process) | cold (after restart) |
|------|--------------|---------------|
| Non-MoE loading | 21s | 25-108s |
| MoE cache loading | 21s | 63s |
| Total startup | ~1:48 | ~3:17-3:51 |

**Verification Environment**:
| Scenario | Startup Time | Non-MoE | MoE | Date |
|------|----------|---------|-----|------|
| TPU VM (chrisya-tpu7-8-02) cold #1 | 3:41 | 108.5s | 63s | 2026-04-22 |

### 22. `MOE_REQUANTIZE_WEIGHT_DTYPE` Omission Causes FP8 cache miss → HBM OOM

**Symptom**: After vLLM startup, log shows all 58 layers MoE "cache miss" (`Cache miss for layer X`), then XLA compilation reports `RESOURCE_EXHAUSTED: CompileTimeHbmOom`, needs 568.87 GB/device but only 94.75 GB available

**Cause**: When `MOE_REQUANTIZE_WEIGHT_DTYPE` env var unset, `fp8.py:_get_config_cache_subdir()` logic defaults dtype to `"fp8"`:

```python
# fp8.py:302
dtype_str = envs.MOE_REQUANTIZE_WEIGHT_DTYPE or "fp8"
```

This causes code to look for `ep8_tp1_gmm_ep_fp8e4m3_bsNone/` subdirectory, while your generated FP4 cache lives in `ep8_tp1_gmm_ep_fp4e2m1_bsNone/`.
All 58 layers cache miss → tries to online requantize from safetensors to FP8 → FP8 MoE weights ~101.8 GB/device exceed 94.75 GB HBM → OOM

**Fix**: Before launching vLLM, **must** set:
```bash
export MOE_REQUANTIZE_WEIGHT_DTYPE=float4_e2m1fn   # Not "fp4", must be full JAX dtype name
```

**Common variant errors**:
- Not set at all → defaults to FP8 → cache miss → OOM
- Set to `"fp4"` → subdir name becomes `..._fp4_...` instead of `..._fp4e2m1_...` → still cache miss
- Set to `"float4"` → same as above

**Lesson**: This is **the easiest to miss and most fatal** environment variable. The other two (`NEW_MODEL_DESIGN=1` and `MOE_WEIGHT_CACHE_DIR`) error out immediately when missing, easy to spot.
But missing `MOE_REQUANTIZE_WEIGHT_DTYPE` causes vLLM to **silently start in FP8 mode** and try to online requantize, until XLA compilation phase OOMs,
and the error message (`CompileTimeHbmOom`) doesn't suggest the env var as root cause, making debugging difficult

### 21. npy_v1 cache meta.json Format Requirements

**Symptom**: On GKE, FP4 cache converted from NPZ to npy_v1 directory format, loading reports `jax.errors.JaxRuntimeError: INVALID_ARGUMENT: Unknown NumPy dtype dtype('V1')`

**Cause**: FP4 (float4_e2m1fn) is stored as `|V1` (1-byte void) dtype in numpy. fp8.py's npy_v1 loading path (`_load_moe_cache_npy_v1()`) depends on `meta.json`'s dtype field to decide whether to do `np_arr.view(ml_dtypes.float4_e2m1fn)` conversion. If meta.json missing these fields, V1 array passed directly to `jax.make_array_from_callback()`, JAX doesn't recognize V1 dtype

**Correct meta.json format**:
```json
{
  "_cache_format": "npy_v1",
  "_storage_format": "native_fp4",
  "w13_weight_dtype": "float4_e2m1fn",
  "w13_weight_scale_dtype": "float32",
  "w2_weight_dtype": "float4_e2m1fn",
  "w2_weight_scale_dtype": "float32"
}
```

**Key fields**:
- `_storage_format: "native_fp4"`: tells loader to use `view(ml_dtypes.float4_e2m1fn)` instead of `view(ml_dtypes.float8_e4m3fn)`
- `w13_weight_dtype` / `w2_weight_dtype`: value containing "float4" triggers dtype view conversion

**Lesson**: `gen_fp4_cache_cpu_parallel.py` correctly generates these fields, but manual NPZ→npy conversion easily misses them. Conversion script must extract dtype info from NPZ metadata and write to meta.json
