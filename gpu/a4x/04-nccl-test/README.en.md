# 5. NCCL Communication Tests

This chapter covers 6 levels of NCCL communication tests: single-node NVLink, intra-domain 2-node MNNVL, cross-domain 2-node RDMA, mixed 4-node, full-domain 18-node 72 GPU, and cross-domain 36-node 144 GPU. Each test includes nominal reference values, measured data from the v1 image, and a reserved column for our own verification.

## 5.1 Single Node 4 GPU (NVLink Baseline)

```bash
# Deploy the single-node test Pod
kubectl apply -f yamls/k8s1341-nccl-single-node.yaml

# Wait for completion
kubectl logs nccl-single-node -f
```

| Metric | Measured Result |
|------|----------|
| all_reduce 4 GPU @8G | **683 GB/s busbw** |

**Single-node NCCL test passed**: all_reduce 683 GB/s busbw (measured on 2026-06-27, consistent).

> **Why does 4 GPU only reach 683 while 8 GPU across nodes reaches 835?** NVSwitch is designed for full interconnect of 72 GPUs. The ring all_reduce over 4 GPUs has only 3 hops and cannot fully utilize all of NVSwitch's parallel channels. When 8 GPUs span 2 nodes, NCCL has more parallel NVLink transfer paths, so busbw is actually higher. Only a full-domain all_reduce over 18 nodes / 72 GPUs can approach the theoretical ceiling of 900 GB/s.

## 5.2 Intra-domain 2-node MNNVL (ComputeDomain + DRANET)

ComputeDomain (already created in [03-gpu-stack](../03-gpu-stack/) step 3.6) manages the IMEX daemon lifecycle. Pods reference a ResourceClaimTemplate via `spec.resourceClaims` to obtain an IMEX channel and enable MNNVL. The core claim pattern (identical to [07-megatron-training](../07-megatron-training/)):

```yaml
# The Pod declares two kinds of resources via DRA:
#   1. compute-domain-channel → ResourceClaimTemplate auto-generated by ComputeDomain (provides the IMEX channel)
#   2. rdma-nics             → manually created ResourceClaimTemplate (provides 4 RDMA NICs)
#
# Key fields in the Pod spec:
#   spec.resourceClaims:
#   - name: compute-domain-channel
#     resourceClaimTemplateName: sd-compute-domain-channel   # ← auto-generated by ComputeDomain in step 3.6
#   - name: rdma-nics
#     resourceClaimTemplateName: rdma-nics-sd-h1             # ← manually defined in the YAML
#   containers[].resources.claims:
#   - name: compute-domain-channel                           # ← container-level reference
#   - name: rdma-nics
```

```bash
# Deploy the intra-domain NCCL test Pods (using ComputeDomain + DRANET)
kubectl apply -f yamls/k8s1341-nccl-same-domain-dranet.yaml

# Wait for the Pods to become ready
kubectl get pods -l name -w

# Verify the IMEX channel is mounted
kubectl exec nccl-sd-h1 -- ls /dev/nvidia-caps-imex-channels/
# Expected: channel0

# Exchange SSH keys (ed25519 is required)
HOST1_KEY=$(kubectl exec nccl-sd-h1 -- cat /root/.ssh/id_ed25519.pub)
HOST2_KEY=$(kubectl exec nccl-sd-h2 -- cat /root/.ssh/id_ed25519.pub)
kubectl exec nccl-sd-h1 -- bash -c "echo '$HOST2_KEY' >> /root/.ssh/authorized_keys"
kubectl exec nccl-sd-h2 -- bash -c "echo '$HOST1_KEY' >> /root/.ssh/authorized_keys"
```

**MPI compilation notes**:

- The `all_reduce_perf` bundled with the pytorch image is **not linked against MPI**; multi-node tests would degrade into independent single-GPU benchmarks, so you must compile the MPI-enabled version from source
- The mpirun path is `/usr/local/mpi/bin/mpirun` (not `/usr/local/gib/bin/mpirun`)
- You must use `-o BatchMode=yes` to avoid interactive SSH prompts

```bash
# Compile the MPI-enabled nccl-tests inside nccl-sd-h1
kubectl exec nccl-sd-h1 -- bash -c "
  cd /tmp && git clone --depth 1 https://github.com/NVIDIA/nccl-tests.git
  cd nccl-tests && make -j8 MPI=1 \
    MPI_HOME=/usr/local/mpi \
    NCCL_HOME=/usr/local/gib \
    CUDA_HOME=/usr/local/cuda
  ldd build/all_reduce_perf | grep libmpi  # should output libmpi.so.40
  cp build/all_reduce_perf /tmp/all_reduce_perf
"

# scp to h2
HOST2_IP=$(kubectl get pod nccl-sd-h2 -o jsonpath='{.status.podIP}')
kubectl exec nccl-sd-h1 -- scp -P 2222 -o StrictHostKeyChecking=no \
  /tmp/all_reduce_perf ${HOST2_IP}:/tmp/all_reduce_perf
```

```bash
# Run the MNNVL test
HOST1_IP=$(kubectl get pod nccl-sd-h1 -o jsonpath='{.status.podIP}')
HOST2_IP=$(kubectl get pod nccl-sd-h2 -o jsonpath='{.status.podIP}')

kubectl exec nccl-sd-h1 -- bash -c "
  source /usr/local/gib/scripts/set_nccl_env.sh 2>/dev/null
  export LD_LIBRARY_PATH=/usr/local/gib/lib64:/usr/local/nvidia/lib64:\$LD_LIBRARY_PATH
  /usr/local/mpi/bin/mpirun --allow-run-as-root \
    -np 8 -npernode 4 \
    --host ${HOST1_IP}:4,${HOST2_IP}:4 \
    -x LD_LIBRARY_PATH -x NCCL_MNNVL_ENABLE=2 -x NCCL_CUMEM_ENABLE=1 \
    --mca plm_rsh_args '-p 2222 -o BatchMode=yes -o StrictHostKeyChecking=no' \
    /tmp/all_reduce_perf -b 512M -e 8G -f 2 -g 1
"
```

> **Pitfall: DRANET DeviceClass must filter out non-RDMA interfaces**. DRANET discovers all network interfaces on the host (including Calico's `vxlan.calico`). A correct DeviceClass must include `device.attributes["dra.net"].rdma == true`; see [03-gpu-stack section 3.4](../03-gpu-stack/).

## 5.4 Mixed 4-node (2 intra-domain + 2 cross-domain)

```bash
# Deploy the 4-node mixed test
kubectl apply -f yamls/k8s1341-nccl-4node-mixed-dranet.yaml

# Wait for all 4 Pods to become ready + exchange SSH keys (pairwise ed25519 public key exchange among the 4 Pods)
# Compile the MPI-enabled nccl-tests and scp to all nodes (same as 5.2)

HOST1_IP=$(kubectl get pod nccl-mix-h1 -o jsonpath='{.status.podIP}')
HOST2_IP=$(kubectl get pod nccl-mix-h2 -o jsonpath='{.status.podIP}')
HOST3_IP=$(kubectl get pod nccl-mix-h3 -o jsonpath='{.status.podIP}')
HOST4_IP=$(kubectl get pod nccl-mix-h4 -o jsonpath='{.status.podIP}')

kubectl exec nccl-mix-h1 -- bash -c "
  source /usr/local/gib/scripts/set_nccl_env.sh 2>/dev/null
  export LD_LIBRARY_PATH=/usr/local/gib/lib64:/usr/local/nvidia/lib64:\$LD_LIBRARY_PATH
  /usr/local/mpi/bin/mpirun --allow-run-as-root \
    -np 16 -npernode 4 \
    --host ${HOST1_IP}:4,${HOST2_IP}:4,${HOST3_IP}:4,${HOST4_IP}:4 \
    -x LD_LIBRARY_PATH -x NCCL_MNNVL_ENABLE=0 -x NCCL_CUMEM_ENABLE=1 \
    --mca plm_rsh_args '-p 2222 -o BatchMode=yes -o StrictHostKeyChecking=no' \
    /tmp/all_reduce_perf -b 1M -e 8G -f 2 -g 1
"
```

**NCCL_MNNVL_ENABLE note**: In mixed-domain tests you must set `NCCL_MNNVL_ENABLE=0`. When set to `2` (auto-detect), NCCL will attempt to probe MNNVL capability on cross-domain nodes (which have no IMEX channel), causing a CUDA error. If you need to leverage both intra-domain MNNVL and cross-domain RDMA simultaneously, you should split the intra-domain and cross-domain nodes into separate communication groups.

**Mixed 4-node test passed**: all_reduce 16 GPU @8G reaches 162.45 GB/s busbw (average 156.7 GB/s). Pure RDMA communication across all 4 nodes works normally.

## Performance Tuning

### NCCL Environment Variables

The GIB diagnostic image ships with the `set_nccl_env.sh` script, which automatically sets the optimal NCCL parameters. Key variables:

| Variable | Recommended Value | Description |
|---|---|---|
| NCCL_NET | gIB | Enable GPUDirect RDMA (GIB) |
| NCCL_MNNVL_ENABLE | 2 (intra-domain) / 0 (cross-domain) | 2 = auto-detect MNNVL, 0 = force disable |
| NCCL_CUMEM_ENABLE | 1 | Enable CUDA Memory Manager |
| NCCL_IB_GID_INDEX | 3 | RoCEv2 GID index |
| NCCL_IB_ADAPTIVE_ROUTING | 1 | Enable adaptive routing |
| NCCL_IB_QPS_PER_CONNECTION | 4 | QPs per connection |
| NCCL_IB_TC | 52 | Traffic Class (DSCP marking) |
| NCCL_PXN_C2C | 1 | Enable PCIe cross-node NVLink relay |
| NCCL_NVLS_ENABLE | 0 | Disable NVLink SHARP (not supported on GB200) |
| NCCL_IB_MERGE_NICS | 0 | Do not merge NICs (independent NIC per GPU) |

### Common Performance Troubleshooting

| Symptom | Possible Cause | Diagnosis |
|---|---|---|
| busbw far below expectation | GIB not enabled | Check `NCCL_NET=gIB`; confirm `gIB` appears in the `NCCL_DEBUG=INFO` logs |
| intra-domain busbw < 800 | MNNVL not enabled | Check `NCCL_MNNVL_ENABLE=2`; confirm the IMEX channel exists (`/dev/nvidia-caps-imex-channels/channel0`) |
| cross-domain CUDA error | MNNVL probe fails on cross-domain nodes | Set `NCCL_MNNVL_ENABLE=0` |
| alltoall extremely slow cross-domain | NCCL chain pollution | Known issue; run alltoall standalone with `k8s134-nccl-36node-2domain-alltoall-sts.yaml` |
| test stuck at barrier | SSH unreachable | Check the sshd port (2222), DNS resolution, and Calico networking |

#### Cross-domain scale effect analysis

**Why is 16n/64GPU much better than 4n/16GPU?**

The core reason is that the **cross-domain traffic share changes differently with scale**, depending on the collective's communication pattern:

**all_reduce / all_gather / reduce_scatter (Ring/Tree algorithms)**:

These three collectives use ring or tree algorithms. Key property: **the number of cross-domain links is fixed at 2 (ring ingress + egress) and does not grow with the GPU count**.

- 4n/16GPU (2+2): the ring has 16 segments, of which 2 go over RDMA = 12.5% cross-domain
  - At the same time, each domain has only 2 nodes × 4 NICs = 8 RDMA links, so aggregate cross-domain bandwidth is limited
- 16n/64GPU (8+8): the ring has 64 segments, of which 2 go over RDMA = 3.1% cross-domain
  - Each domain has 8 nodes × 4 NICs = 32 RDMA links, so aggregate cross-domain bandwidth is 4×

Effect: the slow-link (RDMA) share in the ring drops from 12.5% to 3.1%, and the NVLink segments dominate the pipeline. At the same time, RDMA aggregate bandwidth improves 4×, greatly relieving the bottleneck. all_gather/reduce_scatter recover from 55% → 99%, nearly back to intra-domain levels.

**alltoall (full exchange)**:

alltoall's communication pattern is entirely different: **every GPU must send data to all other GPUs**. Cross-domain traffic = all GPUs in Domain 1 × all GPUs in Domain 2 = N/2 × N/2 = N²/4 pairs.

- 4n/16GPU: 8 × 8 = 64 cross-domain GPU pairs
- 16n/64GPU: 32 × 32 = 1024 cross-domain GPU pairs (16× growth)

RDMA aggregate bandwidth grows only 4×, but cross-domain traffic grows 16×, so the net effect is actually worse. That is why alltoall bandwidth does not improve with scale and stays at ~83-88 GB/s.

**Summary**: ring/tree collectives benefit from scale-out (declining cross-domain link share + rising aggregate bandwidth), while alltoall does not (cross-domain traffic grows O(N²) faster than bandwidth grows O(N)).

#### NCCL Multi-Channel Ring Construction Mechanism

**Doesn't a single ring have only 2 cross-domain links? Then aren't the other nodes' RDMA NICs idle?**

NCCL does not run just one ring; it runs **8-16 channels (rings) simultaneously**, each with a different GPU ordering. The key: **each channel picks a different node pair to do the cross-domain bridging**, thereby spreading cross-domain traffic across all nodes' RDMA NICs.

For example, in an 8+8 node configuration:
- Channel 0: ... d1w0 → **d2w3** → ... → **d2w5** → d1w2 → ... (d1w0/d1w2 do the bridging)
- Channel 1: ... d1w4 → **d2w7** → ... → **d2w1** → d1w6 → ... (d1w4/d1w6 do the bridging)
- Channel 2: ... d1w1 → **d2w0** → ... → **d2w4** → d1w5 → ... (d1w1/d1w5 do the bridging)
- ... and so on

Data is split evenly across multiple channels, and each channel runs its own ring. Different rings use different nodes for cross-domain bridging, so ultimately **all nodes' RDMA NICs are used in rotation**.

**Ring construction algorithm** (source `src/graph/`):
1. **Topology discovery**: scan NVLink/NVSwitch/PCIe/RDMA connections and build a weighted topology graph
2. **Graph search**: DFS + backtracking to find a Hamiltonian ring passing through all GPUs, with the goal of max-min optimization (maximizing the bandwidth of the slowest edge in the ring). It prioritizes high-bandwidth NVSwitch/NVLink edges and places slow RDMA edges only where necessary
3. **Multi-channel load balancing**: subsequent channels deliberately avoid the links heavily used by preceding channels, selecting different node pairs for cross-domain bridging

GB200 NVL72 specifics: within a domain all GPUs are fully interconnected via NVSwitch (bandwidth between any two GPUs is identical), so intra-domain ring ordering does not matter. The core optimization of the algorithm lies in the **allocation of cross-domain bridging points**—distributing cross-domain RDMA traffic evenly across the multiple channels.

You can inspect the actual ring topology with `NCCL_DEBUG=INFO NCCL_DEBUG_SUBSYS=GRAPH`.

#### Channel Count Tuning Experiment (cross-domain 8+8 nodes 64 GPU)

**Hypothesis**: With 32 cross-domain RDMA links (8 nodes × 4 NICs), each ring using 2, the theoretical optimal channel count = 16. Can manually tuning the channel count improve bandwidth?

**Method**: Set `NCCL_MIN_NCHANNELS` / `NCCL_MAX_NCHANNELS` to 8/16/32 respectively, compare against default, and run all_reduce @16G (warmup 20 + iter 50).

| Channel Setting | Actual NVLS nChannels | @16G busbw (GB/s) |
|---|---|---|
| default | 32 | **805.0** |
| 8 | 32 | 802.8 |
| 16 | 32 | 803.1 |
| 32 | 32 | 803.5 |

**Conclusions**:

1. **NCCL already uses 32 channels by default**. Under the GB200 NVSwitch topology, NVLS tuning automatically selects 32 channels, independent of the `NCCL_MIN/MAX_NCHANNELS` environment variables
2. **`NCCL_MIN/MAX_NCHANNELS` has no effect in NVLS mode**. NVLS (NVLink SHARP) has independent channel management logic and is not controlled by the traditional ring channel parameters
3. **The bandwidth difference is < 0.5%** (802.8 ~ 805.0), pure test noise with no statistical significance
4. **The default configuration is already optimal**. NCCL 2.30.4's auto-tuning on the GB200 NVSwitch topology is already optimal, requiring no manual intervention

> **Practical recommendation**: Do not manually set `NCCL_MIN/MAX_NCHANNELS`. NCCL's automatic topology detection and NVLS tuning are already very mature on GB200 NVL72, and manual intervention brings no additional benefit. If you encounter insufficient bandwidth on other topologies (such as H100 PCIe + NVLink Bridge), you may consider tuning the channel count, but the NVSwitch full-interconnect topology needs no tuning.

### Cross-domain NCCL Pitfalls and Troubleshooting Experience (2026-06-29)

Getting cross-domain NCCL from completely non-functional to working went through 3 rounds of debugging:

**Round 1 (failure)**: single StatefulSet + no ComputeDomain channel + `NCCL_MNNVL_ENABLE=0`
- Symptom: NCCL initialization hangs after `NCCL version 2.30.4` with no further output; GPU utilization 0%
- Cause: after the GIB plugin loads, it tries to establish RDMA channels, but the pod has no IMEX channel, so GIB's internal initialization hangs

**Round 2 (works but slow)**: hostNetwork + `NCCL_NET=Socket` + do not load GIB
- Symptom: works, but busbw is only 10.4 GB/s (GVNIC TCP ceiling)
- Significance: proves that cross-domain network routing is fully reachable; the problem is in the GIB plugin initialization

**Round 3 (works correctly)**: dual ComputeDomain + dual StatefulSet + GIB + `NCCL_MNNVL_ENABLE=2`
- Symptom: all_reduce 690 GB/s, all 4 collectives pass
- Key: each pod must have **the ComputeDomain channel of its own domain**

**Root cause summary**: Cross-domain pods seemingly do not need IMEX (inter-domain traffic does not use NVLink), but during initialization the GIB plugin probes all available communication paths, and if a pod has no IMEX channel, GIB hangs while probing the NVLink path. Once each pod is assigned the channel of the correct domain, GIB can correctly distinguish the two paths of "intra-domain NVLink + inter-domain RDMA".

**Deployment key points**:
1. Create an independent ComputeDomain for each domain (`numNodes: 0`)
2. Label each domain's nodes with the corresponding CD's UID label
3. Constrain the two StatefulSets to their respective domains via `nodeSelector: nvidia.com/gpu.clique`
4. Each StatefulSet references its own domain's `resourceClaimTemplateName`
5. `NCCL_MNNVL_ENABLE=2` (do not set 0; let NCCL auto-detect)
6. Allocate RDMA NICs via the DeviceClass `rdma-devices` (which includes the `rdma == true` filter)
