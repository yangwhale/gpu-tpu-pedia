# 8. Megatron-LM Training

## DRA Resource Claim Pattern (Core YAML Structure)

The following is a condensed view of the key DRA-related fields in the training YAML:

```yaml
# 1. ResourceClaimTemplate — declares the need for 4 RDMA NICs (allocated by DRANET)
apiVersion: resource.k8s.io/v1
kind: ResourceClaimTemplate
metadata:
  name: rdma-nics-mega-host-1
spec:
  spec:
    devices:
      requests:
      - name: rdma-nics
        exactly:
          deviceClassName: rdma-devices    # ← DeviceClass, provided by DRANET
          count: 4                         # ← 4 RDMA NICs per A4X node

---
# 2. Pod spec — references ComputeDomain and RDMA via resourceClaims
apiVersion: v1
kind: Pod
metadata:
  name: mega-host-1
spec:
  nodeSelector:
    topology: same-domain                  # ← manually schedule onto the target domain
  containers:
  - name: pytorch
    image: nvcr.io/nvidia/pytorch:26.05-py3
    resources:
      limits:
        nvidia.com/gpu: 4                  # ← GPUs allocated by nvidia-device-plugin
      claims:
      - name: compute-domain-channel       # ← references resourceClaims[0] below
      - name: rdma-nics                    # ← references resourceClaims[1] below

    resourceClaims:                          # ← Pod-level resource claims
    - name: compute-domain-channel
      resourceClaimTemplateName: sd-compute-domain-channel  # ← auto-generated by ComputeDomain
    - name: rdma-nics
      resourceClaimTemplateName: rdma-nics-mega-host-1      # ← manually created template
```

**DRA ≠ scheduling**: The `resourceClaims` above solve "resource allocation"—requesting IMEX channels and RDMA NICs from the K8s DRA framework. But DRA is **not responsible** for scheduling multiple Pods onto the same NVL72 domain.

**Domain-level scheduling** requires a separate mechanism:
- **Manual approach**: `nodeSelector: {topology: same-domain}` (the method used in this chapter, suitable for validation)
- **Production approach**: Kueue TAS's `podset-required-topology` annotation (see [08-multi-domain](../08-multi-domain/))

## NGC Image Rerun State Machine Pitfall (Important)

The Megatron-LM built into the NGC image `megatron-ngc:tev2.15-mgcore_r0.16.0-pt26.05-py3-v2` has a **Rerun State Machine** (a hardware fault detection mechanism) whose default configuration causes FP8 training to exit after only 2 iterations.

**Symptom**: Training starts normally, the logs show `train_iters=50`, but `[after training is done]` appears after the 2nd iteration. There is no error message whatsoever.

**Root cause**: In `megatron/training/resilience_config.py`, the default value of `rerun_mode` is `"validate_results"`. This mode executes forward/backward twice per training step and compares result consistency to detect GPU hardware faults. FP8 training (`--fp8-format hybrid`) is inherently non-deterministic, so the two computations necessarily differ, and the state machine misdiagnoses this as a hardware fault and exits.

**Fix**: Patch the default value in the launch script:

```bash
# Must be executed before torchrun
MEGATRON_DIR=$(find /opt -name "pretrain_gpt.py" -type f 2>/dev/null | head -1 | xargs dirname)
sed -i 's/rerun_mode.*=.*"validate_results"/rerun_mode: Literal["disabled", "validate_results", "report_stats"] = "disabled"/' \
  $MEGATRON_DIR/megatron/training/resilience_config.py
```

> **Note**: The GRPO-related parameters (`--grpo-iterations`, etc.), although also defined in the image's `arguments.py`, are all gated by `perform_rl_step=False` and have no effect in pretraining mode—they are unrelated to this issue.

> **Another pitfall**: The iteration logs (TFLOP/s, elapsed time) are printed via `print_rank_last` to the **last rank**, not rank 0. In multi-node training, you need to check the logs of the worker node (the pod hosting the highest rank) to see the throughput data.

## 8.1 Single-Node Megatron Training (Qwen3 30B-A3B MoE FP8, 50 iterations)

Single node with EP=4 (4-GPU expert parallelism), using the Qwen3 30B-A3B MoE model + TransformerEngine FP8 to fully exploit B200 compute.

**Key configuration**:

- `--transformer-impl transformer_engine` enables TE fused operators (**do not** use `local`)
- `--fp8-format hybrid` + `--fp8-recipe delayed` enable FP8 training
- Environment variables `NVTE_FUSED_ATTN=1` / `NVTE_NORM_*_USE_CUDNN=1` enable GB200-optimized kernels
- `--expert-model-parallel-size 4` (single node: 128 experts / 4 GPUs = 32 experts/GPU)
- `--moe-grouped-gemm` enables grouped GEMM to optimize MoE computation

### Memory Analysis and MBS Calculation

The larger EP is, the fewer experts per GPU, freeing up memory that should be used to increase MBS and improve GPU utilization.

| EP | Experts/GPU | Model footprint (GB) | Available for activation (GB) | Recommended MBS | GBS (GA=4) |

| 4 | 32 | 72.8 | 119 | **16** | 256 |

| 16 | 8 | 22.7 | 169 | **32** | 2048 |
| 32 | 4 | 14.3 | 178 | **32** | 4096 |

> Model footprint = Expert weights (FP8) + Expert optimizer (FP32 AdamW) + Shared weights (BF16) + Shared optimizer (distributed). Activation is estimated at ~3 GB per sample (including recompute). MBS is taken as a power of 2 + 20% headroom.

### Batch Hierarchy Explanation

Megatron-LM's batch is divided into three levels:

| Level | Meaning | Computation |
|---|---|---|
| **Micro Batch Size (MBS)** | Number of samples per forward/backward on a single GPU | Set directly via `--micro-batch-size` |
| **Per-Device Mini Batch** | Total number of samples accumulated per optimizer step on a single GPU | GA × MBS |
| **Global Batch Size (GBS)** | Total number of samples across all GPUs per optimizer step | GA × MBS × DP_effective |

In EP mode, DP_effective = EP (for the attention layers, the EP GPUs are naturally DP).

**GA = GBS ÷ (MBS × EP)**: each GPU accumulates GA micro batches before performing one parameter update.

#### Tokens per GPU per step

| Test | EP | MBS | GA | Per-Device Samples | Per-Device Tokens (seq=4096) |
|---|---|---|---|---|---|
| 1 | 4 | 16 | 256÷(16×4)=4 | 64 | 262,144 |

### Deployment Method

Use hostNetwork mode (bypasses the DRANET GID issue) + ComputeDomain channel (IMEX for MNNVL). For each test group:

1. Create the corresponding number of worker VMs (Domain 2, `forrest-a4x-1x72-policy`)
2. Join the k8s cluster + label + ComputeDomain
3. Deploy the StatefulSet (hostNetwork + ComputeDomain channel)
4. Launch Megatron-LM training with torchrun
5. Run 50 steps, take the average TFLOP/s/GPU over iters 5-44

### Performance Metric Collection

- `s/iter`: training time per step
- `TFLOPS/GPU`: per-GPU throughput
- `MFU`: TFLOPS/GPU ÷ peak compute (GB200 FP8 ~4500 TFLOP/s → MFU% = TFLOPS/GPU ÷ 4500)
- `all-to-all time`: EP communication time fraction (if available from the profiler)

### Benchmark Results v2 (mcore v0.17.0 + standardized methodology)

The old results above used the NGC megatron image (mcore r0.16.0) + a non-standard configuration (full 48-layer model, custom GIB settings), so the results are not comparable. Below are the results re-tested using a standardized benchmark methodology.

**Environment changes**:
- **Image**: `nvcr.io/nvidia/pytorch:26.04-py3` (standard NGC PyTorch, not the Megatron-customized image)
- **Megatron-Core**: v0.17.0 (GitHub `core_v0.17.0` tag, which fixes the r0.16.0 FSDP DTensor bug)
- **Layer count**: 12 layers (standard MoE benchmark methodology, equivalent to the full model after MFU normalization)
- **GIB**: v1.1.2 + `LD_PRELOAD` loading NCCL (including the libibverbs/libmlx5 RDMA libraries)
- **Data**: `--mock-data` (eliminates I/O variables)
- **Sequence length**: 16384

**2-node 8-GPU (same-domain MNNVL) sweep results**:

| Config | EP | MBS | GBS | Recompute | Dtype | TFLOP/s/GPU | MFU (BF16) | HBM Peak (GiB) | Notes |
|---|---|---|---|---|---|---|---|---|---|
| A1 | 8 | 1 | 256 | none | BF16 | **~492** | **21.9%** | 60 | EP=8 baseline |
| A2 | 8 | 2 | 256 | none | BF16 | **~527** | **23.4%** | 102 | **Best configuration** |
| A3 | 8 | 2 | 256 | none | FP8 | **~503** | 22.4% | 93 | FP8 is actually 5% slower (MoE grouped GEMM FP8 overhead) |
| A4 | 8 | 2 | 256 | selective | BF16 | **~480** | 21.3% | 105 | recompute overhead ~9% |
| A5 | 8 | 4 | 256 | selective | BF16 | OOM | — | — | MBS=4 OOMs even with selective recompute enabled |
| A6 | 4 | 1 | 256 | none | BF16 | **~463** | 20.6% | 63 | EP=4 DP=2, trading communication for parallelism |
| A7 | 4 | 2 | 256 | none | BF16 | **~524** | 23.3% | 105 | EP=4 close to the EP=8 best |

> MFU is computed based on the GB200 BF16 peak of 2,250 TFLOP/s. If FP8 MFU is based on the FP8 peak of 4,500 TFLOP/s, it would be 11.2%.

**Key findings**:
1. **MBS=2 is the sweet spot**: MBS=1→2 gives a 7% improvement; MBS=4 OOMs. HBM jumps from 60 GiB to 102 GiB
2. **FP8 provides no speedup for MoE**: The FP8 path of grouped GEMM has overhead that cancels out the Tensor Core acceleration, so BF16 is actually faster
3. **EP=4 vs EP=8 gap is small**: At MBS=2, EP=4 (524 TFLOP/s) is close to EP=8 (527 TFLOP/s), showing that NVSwitch all-to-all is highly efficient
4. **Selective recompute costs ~9%**: Drops from 527 to 480 TFLOP/s, but saves HBM (useful for larger-MBS scenarios)
5. **v0.17.0 vs v0.16.0**: Same config (EP=8 MBS=1 BF16) went from ~105 to ~492 TFLOP/s (4.7×), mainly because the old test was the full 48-layer model + old version

> **B5 OOM analysis**: The OOM at MBS=8 is not MoE dispatch, but the cross-entropy loss's vocab logits tensor [seq=16384, MBS=8, vocab=128256] × FP32 = 62.7 GiB. This is an inherent limit of vocab size × sequence length, unrelated to MoE. Solution: TP>1 to shard the vocab across multiple GPUs.

> B1 vs A1 (12-layer EP=8): 446 vs 492 TFLOP/s, a 9.4% drop. This mainly comes from the increased all-to-all communication overhead of EP=32 (32-way vs 8-way), not from the increased layer count.

**FSDP status**: mcore v0.17.0 fixed the r0.16.0 DTensor `main_grad` bug (`--ckpt-format fsdp_dtensor` must be explicitly specified), but the `--use-megatron-fsdp` + EP=32 combination deadlocks during NCCL initialization. It still hangs even after removing `--use-distributed-optimizer` and the overlap parameters. The 48-layer EP=32 MBS=1 without FSDP runs normally.

**GQA and TP constraints**: Qwen3 30B-A3B uses GQA (Q heads=32, KV heads=4). TP must divide both Q heads and KV heads simultaneously, so the maximum TP is 4. TP and EP are orthogonal (TP shards attention, EP shards MoE), and TP=4 + EP=32 can coexist on 32 GPUs.

### Shared Storage: Lustre Installation (Rocky 9 ARM64 / CIQ kernel)

Multi-node training requires shared storage for the dataset (Megatron's dataset index cache is only built on rank 0; other nodes need to read the same path).

#### Creating a Lustre Instance

```bash
# 1. Create the PSA IP range (if the VPC doesn't have one yet)
gcloud compute addresses create lustre-psa \
  --project=$PROJECT --global \
  --purpose=VPC_PEERING \
  --addresses=10.200.0.0 --prefix-length=16 \
  --network=$VPC_NAME

# 2. Create or update the PSA peering
gcloud services vpc-peerings update \
  --project=$PROJECT --network=$VPC_NAME \
  --ranges=lustre-psa \
  --service=servicenetworking.googleapis.com --force

# 3. Create Lustre (minimum 36000 GiB ≈ 35 TiB)
gcloud lustre instances create $LUSTRE_NAME \
  --project=$PROJECT --location=$ZONE \
  --capacity-gib=36000 \
  --filesystem=lustrefs \
  --network=$VPC_NAME \
  --per-unit-storage-throughput=250
```

#### Installing the Lustre 2.14 Client (Rocky 9 ARM64 pitfalls)

A4X nodes are ARM64 (Grace CPU), and Google's official Lustre client repo has no aarch64 packages. You need to compile from source using DDN's DKMS package, but there are two pitfalls:

1. **OFED conflict**: The nodes come with OFED (Mellanox RDMA) pre-installed, and the Lustre configure detects OFED but errors out because it can't find the devel package. You need `--with-o2ib=no` (Lustre uses TCP and doesn't need RDMA).
2. **DKMS ko2iblnd**: `dkms.conf` lists the `ko2iblnd` module (RDMA driver), but `--with-o2ib=no` skips compiling it, and DKMS errors out because it can't find the .ko file. You must delete the **three lines** for ko2iblnd from `dkms.conf` (NAME + LOCATION + DEST), not two lines.

```bash
# Run on each worker
# Step 1: Install dependencies
dnf install -y dkms
dnf config-manager --set-enabled crb
dnf install -y libyaml-devel json-c-devel
dnf install -y kernel-devel-$(uname -r)

# Step 2: Add the DDN Lustre repo + install the DKMS package
cat > /etc/yum.repos.d/lustre-client.repo << 'EOF'
[lustre-client]
name=GCP Lustre Client
baseurl=https://us-yum.pkg.dev/projects/lustre-client-binaries/lustre-client-rocky-9
enabled=1
gpgcheck=0
repo_gpgcheck=0
EOF
dnf install -y lustre-client-dkms-2.14.0_ddn256-1.el9 lustre-client-2.14.0_ddn256-1.el9

# Step 3: Patch dkms.conf — delete the 3 lines for ko2iblnd (NAME + LOCATION + DEST)
# In the original file these 3 lines come after ksocklnd (around line 31-33):
#   BUILT_MODULE_NAME[...]="ko2iblnd"
#   BUILT_MODULE_LOCATION[...]="lnet/klnds/o2iblnd/"
#   DEST_MODULE_LOCATION[...]="/extra/lnet/"
sed -i '31,33d' /usr/src/lustre-client-2.14.0_ddn256/dkms.conf

# Step 4: Patch configure — disable o2ib
sed -i 's|./configure |./configure --with-o2ib=no |' \
  /usr/src/lustre-client-2.14.0_ddn256/lustre-dkms_pre-build.sh

# Step 5: DKMS build + install
dkms remove lustre-client/2.14.0_ddn256 --all 2>/dev/null || true
dkms add lustre-client/2.14.0_ddn256
dkms build lustre-client/2.14.0_ddn256 -k $(uname -r)
dkms install lustre-client/2.14.0_ddn256 -k $(uname -r)

# Step 6: Load the modules + mount
modprobe lnet && modprobe lustre
lnetctl lnet configure
mkdir -p /mnt/lustre
mount -t lustre <LUSTRE_IP>@tcp:/lustrefs /mnt/lustre
```

> **Key reminder**: The DEST_MODULE_LOCATION line for ko2iblnd in `dkms.conf` is easy to miss (deleting only the NAME + LOCATION lines will cause a `No 'BUILT_MODULE_NAME' directive specified for record #N` error). You must delete all 3 lines.

#### Mounting Lustre in the Training Pod

After the host mounts Lustre, the Pod accesses it via hostPath:

```yaml
volumeMounts:
- { name: lustre, mountPath: /mnt/lustre }
volumes:
- { name: lustre, hostPath: { path: /mnt/lustre, type: Directory } }
```

The dataset is placed in `/mnt/lustre/qwen-datasets/`, shared for read/write across all nodes.

### Memory Issue Records (2026-06-29)

**EP=4 single-node OOM investigation**:

The megatron-ngc image (`tev2.15-mgcore_r0.16.0-pt26.05-py3-v2`) OOMs when running the 128-Expert MoE on GB200:

1. **MBS=16, 48 layers, hidden=4096**: OOM during the model init phase (weights + optimizer > 192 GB)
2. **MBS=4, 12 layers, hidden=2048, 128 Experts**: Model init passes but activation OOM during forward
3. **MBS=1, 12 layers, hidden=2048, 128 Experts**: Same as above, OOMs even at MBS=1
4. **MBS=1, 12 layers, hidden=2048, 64 Experts**: The first step ran at 49.1 TFLOP/s, but the second step OOMs during backward

**Root cause**: On GB200, Transformer Engine falls back to **unfused attention** (`self.unfused_attention`) instead of using Flash Attention / Fused Attention. The attention_probs matrix in unfused attention is O(seq² × heads × batch) = 4096² × 32 × 1 = 2 GB per sample, causing memory to explode.

**Root cause confirmed**: The missing `--bf16` parameter caused the attention QKV inputs to be FP32; neither FlashAttention nor FusedAttention supports FP32 inputs. The TE debug clearly reports:
```
Disabling FlashAttention 2 for unsupported qkv_dtype = torch.float32
Disabling FusedAttention for unsupported qkv_dtype = torch.float32
Selected backend = UnfusedDotProductAttention
```

**Fix**: Add the `--bf16` parameter. FP8 training must be paired with `--bf16` so that the non-FP8 layers (including attention softmax and dropout) use BF16 instead of FP32. After the fix, TE correctly selects FusedAttention (sub-backend 1).

**MBS tuning results** (64 Experts, EP=4, 12 layers, hidden=2048, seq=4096):
- MBS=1: ~178 TFLOP/s/GPU (low tensor core utilization)
- MBS=2: **~475 TFLOP/s/GPU** (+167%, significant compute efficiency improvement)
- MBS=4: OOM (Expert FFN linear_fc2 activation exceeds memory)
- Maximum usable MBS=2, corresponding to MFU=10.6% (based on the GB200 FP8 peak of 4500 TFLOP/s)

> **Impact of MBS=1 vs MBS=16/32**: At MBS=1, the GPU's tensor core utilization is low—the batch dimension of each GEMM is too small, so kernel launch overhead dominates. Increasing MBS enlarges the GEMM batch dimension, significantly improving compute efficiency. MBS=16 is expected to improve TFLOP/s by 30-50% relative to MBS=1. The old baseline (MBS=1) should not be used as a formal reference; only the results from the new tests, with MBS fully utilizing the memory, constitute a valid benchmark.

### Expected Scaling Trend Analysis

Based on DeepEP's scaling data (2n→4n→8n dropping 6%→9%), the expected EP scaling for MoE training:

| Factor | Impact |
|---|---|
| all-to-all communication volume | As EP increases, each GPU needs to communicate with more targets, and the scatter fragments |
| Expert compute volume | As EP increases, the number of experts per GPU decreases, and the per-expert batch shrinks |
| NVSwitch bandwidth | Full mesh within the domain, theoretically independent of EP (confirmed by NCCL same-domain scaling) |
| Attention compute | Unaffected by EP (fully replicated, each GPU computes independently) |

If the drop caused by EP scaling is in the 5-10%/doubling range, it means NVSwitch is effectively amortizing the all-to-all overhead. If the drop exceeds 20%, it means the Expert compute granularity is too small (too few experts per GPU), and the decline in compute efficiency is the main cause.

### Megatron-LM Parameter Changes

### Short Hostname (Required by Megatron)

Megatron-LM uses Gloo for inter-process communication. If the Pod hostname is too long, it triggers a `File name too long` error. Set the `hostname` field in the Pod spec to a short name.

### Multi-Node Hostname Resolution

When a Pod sets a custom `hostname`, NCCL bootstrap and Gloo call `gethostbyname()` to obtain their own IP. If that hostname has no DNS record, resolution fails and causes a **silent hang** (no output after `initialized tensor model parallel`).

**Fix**: Add a hostname→IP mapping in `/etc/hosts`, or use a headless Service + `subdomain` so that k8s DNS registers automatically. You also need to set `NCCL_SOCKET_IFNAME=eth0` and `GLOO_SOCKET_IFNAME=eth0`.

**R10 key findings**:
- `NVTE_CUTEDSL_FUSED_GROUPED_MLP=1` is the single largest optimization (89→284, +219%)
- `perf_plugins.py` automatically sets this environment variable in Slurm mode; torchrun misses it if not set
- cutedsl without CUDA Graph (284) is even faster than TE CUDA Graph without cutedsl (208)
- cutedsl lowers the HBM peak to 175 GiB (0 alloc retries) because the fused kernel reduces intermediate buffers

**Root cause of the 284 vs 936 gap (3.3×)**:
- **A4X HBM = 184 GiB vs DGX-GB200 HBM = 192 GiB**: The 8 GiB gap prevents the full_iteration CUDA Graph from running on A4X. The CUDA Graph eliminates all host overhead (kernel launch + Python scheduling), which has a huge impact on a MoE model's architecture of many small kernels
- **full_iteration CUDA Graph + MoE flex dispatcher compatibility**: Even when recompute frees up memory (0 alloc retries), CUDA Graph capture still crashes with "stream has unjoined work". The asynchronous dispatch of the MoE token-dropless mode is incompatible with the static capture of a CUDA Graph
- **cutedsl + TE CUDA Graph combination crash**: The Triton kernel produces a "Pointer argument cannot be accessed from Triton (cpu tensor?)" error during CUDA Graph replay—this is a known Triton/CUDA Graph compatibility issue
